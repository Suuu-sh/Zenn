# はじめに

Codex はかなり優秀です。  
ただ、優秀すぎるがゆえに、返答があまりに無機質で  
**「便利なツールではあるけど、相棒感はない」**  
と感じることがありました。

この記事は、

- 誰でも5分で再現できる手順書
- 初心者向けの完全ハンズオン

ではありません。

どちらかというと、  
**どういうプロセスで運用しているか** を整理して残すための構成メモです。

---

# 先に全体像

この記事でやっていることはかなり単純で、  
**Codex の作業フェーズに応じてフレーズキーを選び、そのキーに対応する定型文を VOICEVOX で喋らせる**  
という構成です。

流れを一行で書くとこうです。

```text
ユーザーの依頼
  -> Codexが返答・作業を進める
  -> 作業開始 / 調査中 / ファイル操作 / 問題発見 / 完了 のタイミングで定型文を選ぶ
  -> VOICEVOX Engine で音声化
  -> macOSで再生
```

---

# 1. どういうプロセスでやっているか

今の運用では、ずんだもん化のプロセスは次のようになっています。

## 返答そのもの

返答本文そのものは、普通に Codex が生成します。  
今回やりたかったのは本文朗読ではなく、  
**作業中の状態をイベント音声として扱うこと** でした。

なので、この構成では

- 返答本文
- 状態通知としての音声

を分けて考えています。

---

## 作業中の音声

作業中は、内容全文を毎回読ませるのではなく、  
**イベントごとの定型句** を鳴らしています。

たとえば、

- `work_start`
- `investigating`
- `file_operation`
- `issue_found`
- `task_complete`
- `stopped`

のようなキーを持たせて、  
そのイベントに対応するフレーズだけを喋らせています。

つまり「今何をしているか」を伝えるための音声であって、  
毎回レスポンス全文を朗読する構成ではありません。

---

---

## 完了時の音声

完了後も、返答本文は読みません。  
ここはかなり重要で、長文をそのまま読むと疲れるからです。

完了時は、

- 正常系なら `task_complete`
- 問題発見や失敗寄りなら `issue_found`

というように分類して、  
最後は固定フレーズだけを鳴らしています。

---

# この構成にしてよかったこと

やってみてよかったのは、  
ただ音が鳴るのではなく **状態が耳で分かる** ようになったことです。

たとえば、

- いま調査に入った
- いまファイルを触る
- いま問題が見つかった
- いま完了した

が、短いずんだもんフレーズで分かるようになります。

しかも、全文朗読ではないのでしつこくなりません。

それに加えて、  
返答文体も少しずんだもん寄りになっているので、  
ただのツールというより  
**「変な声の賢い相棒」** に近い感じになります。

最初の動機だった
**「すごく賢いのにアホ声」**
というギャップは、思ったより効きました。

---

# まとめ

この構成でやっていることをまとめると、次の通りです。

- Codex が返答・作業を進める
- 作業開始 / 調査中 / ファイル操作 / 問題発見 / 完了 / 停止 のタイミングで定型キーを選ぶ
- そのキーに対応するずんだもんフレーズだけを鳴らす

つまり、
**イベント・定型文・音声合成を分離して組んでいる**
というのがこの構成の本質です。

「誰でも簡単に」よりも、  
**自分が納得できる相棒感をどう作るか**
に寄せたかったので、今のところかなり満足しています。
