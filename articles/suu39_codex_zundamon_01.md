# はじめに

Codex は優秀ですが、返答が無機質で  
**「便利なツールではあるけど、相棒感はない」**  
と感じることがありました。

そこで、
**ものすごく賢いのに、アホ声で状況報告してくる相棒**
みたいな存在に寄せられないかと思って、
Codex と VOICEVOX をつないで、ずんだもん化してみました。

この記事は
**誰でも5分で再現できる手順書**
ではなく、
**いま自分がどういうプロセスで運用しているか**
を整理して紹介するための記事です。

---

# 先に全体像

やっていることは、ざっくり 2 本立てです。

- 作業中は、フェーズごとの**定型フレーズ**を鳴らす
- 完了時は、Codex が考えた**短い要約文**を喋らせる

流れを一行で書くとこうです。

```text
ユーザーの依頼
  -> Codexが返答・作業を進める
  -> 作業開始 / 調査中 / ファイル操作 / 問題発見 / 停止 では定型フレーズキーを選ぶ
  -> 完了直前だけは task_complete / issue_found と短い要約文をキューする
  -> notify hook がキューを回収する
  -> VOICEVOX Engine で音声化
  -> macOSの通知センター + 音声再生
```

最初は完了時も固定フレーズだけでしたが、  
今は
**「何が終わったのか」まで短く喋る**
構成に変えています。

たとえば、

- `開始時に完了通知が来るバグを直したのだ`
- `Appleサインインの設定で未確認箇所が残っているのだ`

のような一文を、Codex が完了時に生成して通知させています。

# 1. どういうプロセスでやっているか

今の運用では、ずんだもん化のプロセスは次のようになっています。

## 返答そのもの

返答本文そのものは、普通に Codex が生成します。  
今回やりたかったのは本文朗読ではなく、  
**作業中の状態をイベント音声として扱うこと** でした。

なので、この構成では

- 返答本文
- 状態通知としての音声

を分けて考えています。

---

## 作業中の音声

作業中は、返答内容全文を毎回読ませるのではなく、  
**イベントごとの定型句** を鳴らしています。

たとえば、

- `work_start`
- `investigating`
- `file_operation`
- `issue_found`
- `stopped`

のようなキーを持たせて、  
そのイベントに対応するフレーズだけを喋らせています。

つまり「今何をしているか」を伝えるための音声であって、  
毎回レスポンス全文を朗読する構成ではありません。

---

## 完了時の音声

ここが最初の構成から変わったところです。  

いまは完了後も返答全文は読まず、  
**Codex が考えた短い要約文だけ** を喋らせています。

たとえば完了前に、

```bash
~/.codex/bin/voicevox-queue-next-phrase.sh task_complete "開始時に完了通知が来るバグを直したのだ"
```

のように、

- 通知の種別
- 実際に喋らせる一文

を先にキューしておきます。

そのあと返答が終わると `notify` が走って、
キュー済みの内容を回収し、

- macOS の通知センター
- 通知音
- VOICEVOX の読み上げ

をまとめて流す構成にしています。

ここでの `task_complete` / `issue_found` は、
主に

- 成功通知か問題通知かの種別
- 通知センターのタイトル分岐
- 要約文がない場合の fallback

として使っています。

つまり、今は
**完了通知の本文そのものは固定文ではなく、Codex がその場で短く考えた一文**
になっています。

# この構成にしてよかったこと

やってみてよかったのは、  
ただ音が鳴るのではなく **状態が耳で分かる** ようになったことです。

たとえば、

- いま調査に入った
- いまファイルを触る
- いま問題が見つかった
- いま完了した

が、短いずんだもん音声で分かるようになります。

しかも完了時は単なる
`完了したのだ`
ではなく、

- `開始時に完了通知が来るバグを直したのだ`

のように、
**何が終わったかまで音で入ってくる**
ので、他の作業をしながらでも状況を把握しやすくなりました。

普通の通知でも完了は分かりますが、
そこがずんだもんになっているだけで妙に愛着が湧きます。

それに加えて、  
無機質なツールというより  
**「アホ声の賢い相棒」** に近い感じになります笑

**「すごく賢いのにアホ声」**
というギャップが愛おしく、開発のモチベにもなり良かったです。

# まとめ

この構成でやっていることをまとめると、次の通りです。

- Codex が返答・作業を進める
- 作業開始 / 調査中 / ファイル操作 / 問題発見 / 停止 では定型キーを選ぶ
- 完了時だけは `task_complete` / `issue_found` と短い要約文を Codex がキューする
- notify hook がそれを回収して、通知センターと VOICEVOX 再生につなぐ

つまり、
**イベント・定型文・完了要約・音声合成を分離して組んでいる**
というのがこの構成の本質です。

「誰でも簡単に」よりも、  
**自分が納得できる相棒感をどう作るか**
に寄せたかったので、今のところかなり満足しています。
